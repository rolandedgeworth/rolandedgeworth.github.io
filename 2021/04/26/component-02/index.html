VueCli Level 1：Component 组件使用

发表于 2021-04-26 分类于 VueCli ， components

本章讲解组件（component）相关的基本使用方式

组件的引用

假设一个 list.vue 组件，这是一个列表性质的页面

在这个示例中，我将 script 标签放到了最上面，一般我们并不这样写，但为了便于理解，在这里使用这样的方式进行表达

示例中已使用注释进行了说明，一个组件的使用分为三步：引入、注册和使用，值得注意的是第二步，import 引入后，需要在 components 中进行注册，示例中的 “components: {listItem}” 是一种简写，全写应为 “components: {listItem: listItem}”，左侧的 listItem 是键名，右侧的 ListItem 是取值，它在第一步（import）时被声明

另外，当一个组件被注册后，就可以在 template 中使用了，使用时可以将注册时使用的驼峰命名转为短横线命名，<ListItem> 或 <list-item>，没有区别

组件的封装

页面性质

日常开发工作中，我们接触最多的页面有三大类：列表、表单、详情

在功能上，它们有着本质上的区别，但从开发角度讲，这些页面其实十分相似，它们是一个容器，或者称为平台，它们存储主要（也是绝大多数的）数据，它们容纳功能性组件，并将它们限制在一个独立的空间内

在我的理解内，这一类型的页面应该是这样的：

一个容器，包裹着各种或大或小的功能，为我们开发时提供一个缓冲，避免一个文件千行代码的情况出现
当然，对于某些功能十分简单的页面，就不必将其再次拆分，直接将页面与功能结合到一个组件中即可

功能性质

本文所提到的功能型组件，是指那些提供某种业务性功能的组件，大到多步式表单的某一页，小到一个基本的输入框，当你在开发时感到某一页状态太多，方法太繁杂，不要犹豫，将它拆开，即使拆分后的组件毫无复用性，即使拆分后这一页可能只剩寥寥几十行代码，也要拆开，否则当三五个月之后，Leader 叫你修改某项功能时，你很可能不得不对自己的代码进行重构

页面型组件可以较为单一，但功能型组件必然是极具多样性的，它们要涵盖产品经理提出的一切花样

一般来说，项目初期，甚至在开始前，就要有一两个人去关注产品需求，去逐页查看 UI 设计，他们要将那些细小却随处可见的基础型功能抽离，封装为一个个全局组件，即使使用了 UI 框架进行开发，样式上的大幅度修改，或基于框架进行二次封装，也是必不可少的环节。这一过程基本完成后，其余开发人员介入，进行具体业务性质的开发，基于一个个全局的、高复用性的组件，再次封装局部组件

举个例子：我在全局封装了一个输入框，”my-input“ 组件，之后，我封装了一个包含十个输入项的 “base-form” 组件，和一个包含十五个输入项的 “educational-form” 组件，最后将这两个 form 组件放进了一个名为 “personalInfoEdit” 的页面组件中

这里的 “educational-form” 组件和 “base-form” 组件，从功能来看，其代码必然是复杂且冗长的，同时它们大概率只在 personalInfoEdit，即个人信息编辑页面中有应用，不会再被第二个模块所需要，但对它们进行抽离，将其独立为一个组件，仍然是有价值、有意义的，这样做
